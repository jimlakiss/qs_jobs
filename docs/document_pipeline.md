# Document Pipeline

This note captures the intended direction for project documents, extracted data, audit history, and future permissions.

## Principle

Documents and extracted records should generally not be hard-deleted in normal use. They should move through states such as current, superseded, archived, or deleted.

Hard deletion can remain an owner/admin maintenance action, but day-to-day workflow should preserve history so project records stay auditable.

## Current Lanes

### Imported Documents

Raw files received for a project.

Examples:
- client email attachments
- consultant drawings
- approvals
- correspondence
- internal uploads

Important metadata:
- received date/time
- source
- received from
- notes
- uploaded by, later
- status, later

### Extracted Data

Structured JSON/table data created by the PDF viewer extraction workflow.

This is project intelligence rather than just another file.

Important metadata:
- source imported document
- extraction date/time
- extracted by, later
- sheet count
- extraction status
- confirmation/review status, later

### Extracted Documents

Curated files generated by the viewer.

Examples:
- renamed drawing ZIPs
- CSV manifests
- JSON manifests
- later, split PDFs

Important metadata:
- generated from source imported document
- generated from extraction record
- export kind
- generated date/time
- generated by, later

## Lifecycle States

Recommended future status values for documents and extraction records:

- `current`
- `superseded`
- `archived`
- `deleted`

The `deleted` state should mean hidden from normal views, not necessarily removed from storage.

Useful future fields:

- `status`
- `superseded_by_id`
- `archived_at`
- `deleted_at`
- `deleted_by_id`
- `delete_reason`
- `created_by_id`
- `updated_by_id`

## Deletion Policy

Preferred behavior:

- Imported/source documents are preserved unless an owner performs a hard delete.
- Deleting or archiving an extraction should also hide/archive the extracted documents generated from that extraction.
- Extracted documents should remain linked to both the source document and the exact extraction record that created them.
- The app should avoid leaving multiple visible versions of the same derived output without marking which is current.

## Permission Direction

Future roles can build on the same lifecycle and audit model.

Suggested tiers:

- Owner/admin: full access, including hard delete and user/permission management.
- Manager/internal user: create and edit jobs, upload documents, run extraction, mark records current/superseded, but no hard delete.
- Document contributor: upload documents to existing jobs only.
- Client: upload documents to assigned projects, view only released/allowed documents.
- Read-only/client released: view completed/released documents only.

Permissions should be checked against both:

- the user's role
- the document or project state

Examples:

- Clients can upload into Imported Documents but cannot see Internal or unreleased Extracted Documents.
- Only owner/admin can hard delete.
- Managers can mark extracted data as superseded but cannot erase audit history.
- Released project documents become visible to clients after payment/approval workflow later.

## Implementation Notes

Short-term model:

- Keep using Active Storage for files.
- Keep `project_documents` as the metadata layer around attachments.
- Keep `document_extractions` for structured extraction JSON.
- Link extracted documents to the exact extraction record that generated them.

Future model upgrades:

- Replace hard deletes with soft-delete/status transitions.
- Add actor fields once employee/client accounts exist.
- Add an audit event table for important transitions.
- Add filters for current/superseded/archived/deleted views.

## Audit Event Ideas

Potential future event names:

- `document_uploaded`
- `document_metadata_updated`
- `document_superseded`
- `document_archived`
- `document_deleted`
- `extraction_created`
- `extraction_superseded`
- `extraction_deleted`
- `export_generated`
- `client_access_granted`
- `client_access_revoked`

Each event should store:

- project
- actor
- target record
- event name
- timestamp
- before/after metadata where useful
